## 프레임 예산과 실행 조건

앞선 글에서는 같은 장면을 다시 열기 위해 무엇을 기록해야 하는지 썼다.
world state만 저장하는 수준을 넘어, 한 프레임 안에서 어떤 후보가 생겼고, 어떤 순서로 계산이 이어졌는지도 같이 붙잡아야 한다는 이야기였다.

그 다음에 바로 성능 문제가 따라왔다.

같은 장면을 다시 여는 물리 시뮬레이션에는 고정된 실행 조건이 꽤 많이 필요하다.
숫자 규칙, 작업 순서, 결과를 합치는 순서, 중간 산출물의 이름 같은 것들이다.
물리 시뮬레이션은 매 프레임 돈다.
60fps만 잡아도 한 프레임 예산은 약 `16.6ms`다.
렌더링, 입력, 게임 로직까지 같이 생각하면 물리 쪽에서 쓸 수 있는 시간은 더 짧다.

여기서 생각이 멈췄다.

재현성을 위해 기록과 검증을 늘리면 실행이 무거워진다.
성능을 보려고 실행을 backend에 넓게 맡기면 같은 장면을 다시 비교할 근거가 흐려진다.

내가 원한 것은 둘을 같이 잡는 실행 방식이었다.
같은 입력에서 같은 결과가 나오고, 원하는 지점부터 다시 열 수 있고, 그 상태에서도 프레임 예산 안에 들어와야 했다.

이 글에선 그 과정에서 본 성능 개선 작업 과정을 나열하고자 한다.

## 연산량, 메모리, 할당

성능을 생각하면 먼저 연산량을 떠올린다.
곱셈을 몇 번 하는지, 충돌 검사를 몇 번 하는지, 루프가 몇 번 도는지 같은 것들이다.

실제로는 계산 밖에서도 시간이 빠진다.
메모리에서 데이터를 가져오는 비용, 새 공간을 할당하는 비용, 분기 때문에 CPU 예측이 꼬이는 비용, 작은 작업을 너무 많이 쪼개며 생기는 비용이 같이 붙는다.

물리 시뮬레이션에서는 이 비용이 꽤 잘 보였다.
한 프레임 안에서 대략 다음 일이 계속 반복된다.

```text
이전 상태
  -> 충돌 가능성이 있는 쌍 찾기
  -> 실제 접촉 정보 만들기
  -> 제약 조건 만들기
  -> solver 반복
  -> 위치와 속도 갱신
  -> 다음 상태
```

밖에서 보면 `step()` 한 번이다.
안쪽에서는 여러 종류의 데이터가 계속 만들어지고, 정렬되고, 다시 합쳐진다.

재현 가능한 실행에서는 이 중간 과정도 흔들리면 곤란했다.
같은 접촉 후보가 같은 이름으로 나오고, 같은 순서로 solver에 들어가고, 같은 규칙으로 결과가 합쳐져야 한다.
성능을 올린다고 이 순서가 흔들리면, 다음에 같은 장면을 열었을 때 비교할 기준이 약해진다.

그래서 성능을 볼 때 기준을 세 개로 나눴다.

```text
빠른가
  median 시간
  p95 시간

프레임 중 흔들림이 적은가
  새 할당 수
  메모리 사용량
  실행 중 capacity 초과 여부

다시 열 수 있는가
  replay hash
  같은 입력에서 같은 결과
```

평균 시간이 좋아져도 p95가 튀면 실제 프레임에서는 버벅임으로 보였다.
시간이 줄어도 실행 중 할당이 생기면 어느 프레임에서 갑자기 튈 여지가 생긴다.
replay hash가 달라지면 성능 개선을 했는지, 실행 조건이 바뀐 것인지 비교 기준이 흐려진다.

벤치는 숫자 하나를 얻는 도구보다, 실행 방식을 고르는 기준에 가까웠다.

## Hot, Prepared, Cold 경로

처음으로 한 일은 실행 경로를 세 갈래로 나누는 일이었다.

매 프레임 반드시 도는 코드가 있다.
미리 준비해두면 프레임 안에서 가볍게 쓸 수 있는 코드가 있다.
장면을 만들거나 디버깅할 때만 도는 코드가 있다.

이 셋을 같은 기준으로 보면 금방 꼬인다.
매 프레임 도는 곳에서는 새 할당 하나도 부담이 된다.
반대로 장면을 불러오거나 debug 정보를 뽑는 곳까지 매 프레임 기준으로 묶으면 코드가 괜히 빡빡해졌다.

그래서 기준을 이렇게 나눴다.

```text
hot
  매 프레임 반복 실행
  새 할당 0
  정해진 순서로 결과 확정
  replay hash 유지

prepared
  프레임 전에 미리 준비
  정렬된 입력
  재사용 가능한 작업 공간
  hot에서 바로 읽을 수 있는 형태

cold
  장면 생성, import, debug, 원시 snapshot 처리
  비용은 측정
  매 프레임 경로와 분리
```

이 분리는 성능에도 필요했고, 재현성에도 필요했다.

hot 경로가 작아지면 그 안에서 무엇이 결과를 바꾸는지 보기 쉬워진다.
prepared 경로가 따로 있으면 정렬, 압축, capacity 확인 같은 일을 프레임 전에 치를 수 있다.
cold 경로가 분리되면 debug와 authoring 편의를 살리면서도 프레임 예산을 건드리는 일을 줄인다.

## 캐시 라인과 데이터 배치

그다음에 본 것은 캐시였다.

CPU는 보통 낱개 값보다 cache line 단위로 주변 값을 같이 가져온다.
그래서 자주 쓰는 값들이 메모리에서 가까이 붙어 있으면 유리하다.
반대로 큰 객체를 따라다니며 여기저기 흩어진 필드를 읽으면, 실제 계산보다 데이터를 가져오는 시간이 커진다.

물리 엔진에서 body 하나를 생각해보면, 그 안에는 많은 정보가 들어간다.
위치, 회전, 속도, 질량, collider 연결, material, debug 상태, user data 같은 것들이다.
solver가 매번 읽는 값은 그중 일부다.

객체 단위로 읽으면 다음과 같다.

```text
body[0] = 위치, 속도, 질량, collider, material, debug, ...
body[1] = 위치, 속도, 질량, collider, material, debug, ...
body[2] = 위치, 속도, 질량, collider, material, debug, ...
```

solver가 원하는 값만 따로 모으면 모양이 달라진다.

```text
position_x: [ ... ]
position_y: [ ... ]
position_z: [ ... ]
velocity_x: [ ... ]
velocity_y: [ ... ]
velocity_z: [ ... ]
inverse_mass: [ ... ]
```

두 번째 모양은 사람이 보기에는 조금 덜 자연스럽다.
그만큼 CPU가 연속된 메모리를 훑기 좋다.
SIMD도 이 모양에서 더 잘 붙는다.
한 번에 여러 body의 `x` 값을 가져와 같은 연산을 태우기 좋기 때문이다.

hot 경로가 만지는 데이터량을 줄이자 수치도 바로 움직였다.
아래 값은 한 로컬 측정에서 프레임 중 실제로 만지는 hot 데이터량을 대략 합산한 기록이다.

| Scene | Before | After | Change |
| --- | ---: | ---: | ---: |
| `sparse_256` | `3,317,760` bytes | `184,320` bytes | `-94.4%` |
| `overlap_256` | `4,061,040` bytes | `913,920` bytes | `-77.5%` |
| `dense_96` | `989,360` bytes | `542,720` bytes | `-45.1%` |

여기서 본 것은 bytes 자체보다 방향이었다.
매 프레임 읽고 쓰는 표면을 줄이면 캐시가 버티기 쉬워지고, p95도 같이 안정될 여지가 생긴다.

## SIMD 적용 범위

SIMD는 여러 값을 한 번에 계산하는 방식이다.
예를 들어 정수 네 개를 한 번에 더하거나, 여러 좌표의 같은 축을 한 번에 처리한다.
성능 글에서 자주 나오는 도구지만, 물리 solver에서는 붙이는 위치를 골라야 했다.

이유는 순서였다.

어떤 계산은 서로 독립적이다.
예를 들어 여러 body의 위치를 같은 규칙으로 업데이트하는 작업은 묶기 좋다.
충돌 후보의 bounding box를 같은 방식으로 검사하는 작업도 묶기 좋다.

반대로 solver의 일부 구간은 앞선 결과가 다음 계산의 입력이 된다.
contact A에서 계산한 impulse가 body 속도를 바꾸고, 그 값이 contact B 계산에 들어간다.
이 구간에서는 순서가 물리 의미를 갖는다.

그래서 SIMD를 넣은 기준은 단순했다.

```text
SIMD가 잘 맞는 곳
  같은 모양의 데이터를 연속으로 읽는 곳
  각 항목이 독립적으로 계산되는 곳
  tail 처리 규칙을 고정할 수 있는 곳
  결과를 정해진 순서로 다시 쓸 수 있는 곳

순서를 보존한 곳
  solver 반복처럼 앞 결과가 다음 입력이 되는 곳
  누적값을 어떤 순서로 접는지가 결과에 영향을 주는 곳
```

SIMD는 데이터 모양을 바꾼 뒤에 붙였다.
먼저 데이터를 연속된 column으로 만들고, 처리 단위를 고정하고, 마지막에 결과를 쓰는 순서를 정했다.
그다음에 batch 연산을 태웠다.

수학 벤치에서도 이 순서가 맞았다.
대표적인 fixed-point 연산 몇 개는 손실 0으로 통과했다.

## 벤치마크 기반 의사결정

벤치를 보면서 가장 많이 잘라낸 것은 불필요한 복사와 반복 스캔이었다.

처음에는 안전하게 중간 벡터를 자주 만들었다.
후보를 만들고, 다시 복사하고, 정렬하고, public output으로 또 옮기는 식이다.
디버깅할 때는 편했다.
매 프레임 반복되니 비용이 커졌다.

자주 나온 패턴은 다음 쪽이었다.

```text
복사 줄이기
  중간 버퍼를 줄이고 caller output에 바로 쓰기
  이미 준비한 stream 재사용

반복 스캔 줄이기
  id -> index lookup 유지
  매번 전체 배열 훑기 줄이기
  미리 계산한 bounds 전달

중복 계산 줄이기
  ray length 재사용
  static mesh edge / normal 재사용
  snapshot 검증용 hash 재계산 줄이기

경로 분리
  debug readback은 프레임 밖으로
  contact detail은 진단용으로
  매 프레임 solver는 compact core만 읽기
```

이 과정에서 병목 위치가 계속 바뀌었다.
한 번은 workspace를 조금씩 키우는 과정에서 새 할당이 반복됐다.
한 번은 빈 world에서도 query 준비 경로가 열렸다.
또 한 번은 이미 계산한 bounds를 뒤에서 다시 만들고 있었다.

하나씩 줄였다.
그리고 매번 같은 조건으로 다시 돌렸다.
결과 hash가 같은지, 실행 중 새 할당이 생겼는지, median과 p95가 같이 움직였는지 봤다.

## MVP 성능 지표

아래 수치는 로컬 측정값이다.
플랫폼 전체 성능 약속보다, 실행 방식을 바꿨을 때 어떤 방향으로 움직였는지 보는 기록에 가깝다.

비교 기준은 2026년 4월 6일 메모리 복사 정리 직후 기록과 2026년 4월 14일 측정표다.

| 항목 | Before | After | 대략 변화 |
| --- | ---: | ---: | ---: |
| static mesh 10k + dynamic body 1k world build | `4.15 ms` | `1.77 ms` | `2.34x` |
| batch raycasts mixed 10k | `13.46 ms` | `5.87 ms` | `2.29x` |
| batch raycasts non-overflow 10k | `12.58 ms` | `5.72 ms` | `2.20x` |
| character moves over static mesh + terrain | `2.58 ms` | `1.14 ms` | `2.26x` |
| vehicle tire contacts over terrain | `0.131 ms` | `0.055 ms` | `2.40x` |
| prepared snapshot write only | `0.902 ms` | `0.639 ms` | `1.41x` |

solver 쪽에서도 변화가 있었다.
contact가 많은 장면에서 동기화와 merge 비용이 줄었다.

| Scene | Metric | Before | After | Change |
| --- | --- | ---: | ---: | ---: |
| `overlap_256` | total | `1.177 ms` | `0.985 ms` | `-16.27%` |
| `overlap_256` | solver sync | `0.376 ms` | `0.209 ms` | `-44.27%` |
| `overlap_256` | contact merge | `0.037 ms` | `0.019 ms` | `-49.66%` |
| `dense_96` | total | `0.920 ms` | `0.815 ms` | `-11.42%` |
| `dense_96` | solver sync | `0.243 ms` | `0.139 ms` | `-42.90%` |
| `dense_96` | contact merge | `0.018 ms` | `0.005 ms` | `-71.07%` |

작업량이 거의 0에 가까운 프레임도 따로 봤다.
충돌 후보가 0개인 장면에서 전체 pipeline을 열어버리면, 실제 작업량은 작아도 프레임 비용이 남는다.
empty-pair scene은 `93,708 ns`에서 `23,292 ns`까지 내려왔다.
대략 4배다.

이 개선은 큰 알고리즘 하나에서 나온 결과라기보다, 할 일이 적은 순간을 빨리 알아차리고 짧게 빠져나오는 처리에서 나왔다.

## 제외된 최적화 후보

측정값 때문에 제외한 시도도 있었다.

정적 mesh query에서는 BVH도 시험했다.
hit hash는 맞았고, hot allocation도 0이었다.
일부 miss-ray 항목에서는 더 빨랐다.
여러 장면에서 p95가 커졌다.
후보 참조 수가 줄어든 장면에서도 실제 시간은 나빠졌다.

그래서 실제 실행 경로에서 제외하고 측정 기록으로만 남겼다.

```text
BVH 실험
  hash: 일치
  hot allocation: 0
  miss-ray: 더 빠름
  여러 장면의 p95: 더 느림
  처리: 측정 기록으로만 남김
```

분기를 넣은 scalar shortcut도 비슷했다.
겉으로는 계산을 줄일 것처럼 보였다.
실제로는 코드 경로가 넓어지면서 median이나 p95가 나빠졌다.
후보 벡터를 건너뛰고 바로 처리 루프로 흘리는 시도도 같은 이유로 빠졌다.

여기서 기준이 조금 분명해졌다.
성능 작업에서는 "이론상 덜 한다"보다 "측정값에서 덜 흔들린다"를 더 믿게 됐다.
특히 replay를 전제로 한 물리 코어에서는 hash, allocation, p95가 같이 맞아야 실제 경로로 올릴 수 있다.

## 재현성, 성능, 반복 실험

이 글에서 적고 싶었던 것은 복잡한 엔진 구조보다 실행 기준이었다.

재현 가능한 물리 시뮬레이션을 빠르게 만들려면, 먼저 실행 경로를 나눠야 한다.
매 프레임 도는 곳은 작게 만든다.
미리 준비할 수 있는 일은 프레임 전에 치른다.
장면 생성이나 debug처럼 무거운 일은 프레임 밖으로 뺀다.

그다음 데이터 모양을 바꾼다.
큰 객체를 매번 훑는 방식을 줄이고, hot 경로가 필요한 값만 연속된 column으로 모은다.
그 위에서 SIMD가 먹히는 곳을 고른다.
순서가 물리 의미를 갖는 곳은 순서를 지키고, 독립적인 반복만 묶는다.

이렇게 하다 보니 결정론과 성능이 같은 경계 위에서 만났다.
결정론을 지키려고 세운 경계가 성능을 손댈 위치도 더 분명하게 만들었다.

계속 성능이 개선되고는 있는데, 점점 개선 폭이 줄어들기 시작했다. 사실 지금 로컬에서만 개선되는 것이라 다른 운영체제나 cpu등에서는 다를 수 있을거라는 생각이 든다. 다양한 플랫폼에서도 테스트해보면서, 일관되게 상승되는지를 확인할 수 있는 테스트 루프를 자동화하는 과정을 거쳐 로컬 뿐만 아니라 다른 환경에서도 일관되게 증가하는지 검증하는 루프를 만드는 것이 또 중요하다고 생각이 든다. 그런데 그게 다 돈이다보니 걱정이 이만저만이 아니다... 헤츠너 클라우드에 서버 옥션으로 빌리거나 하는 등을 모색 중인데, 차차 더 알아봐야겠다.
